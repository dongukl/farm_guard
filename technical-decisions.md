# 설계 판단 기록 — FarmGuard 알고리즘

`designator.py`를 중심으로 한 핵심 설계 결정들과 그 이유를 기록합니다.
각 결정은 "왜 그렇게 했는가"에 초점을 맞춥니다.

---

## 1. 양몰이(Herding) 알고리즘: Strömbom 모델 기반 호(Arc) 플랭킹

### 배경

동물을 특정 방향으로 몰아내는 문제는 단순히 "동물 뒤에서 밀기"로 해결되지 않습니다.
정면에서 밀면 동물이 좌우로 도망가고, 한 쪽에서만 몰면 반대쪽으로 빠져나갑니다.
2014년 Strömbom et al.이 발표한 양몰이 모델은 이 문제를 수학적으로 분석한 연구로,
"몰이꾼이 동물을 목표 지점까지 몰아내는 최소 조건"을 제시합니다.

### 알고리즘 핵심 아이디어

두 로봇이 동물을 정면에서 미는 대신, **게이트 반대쪽에서 좌우로 벌어져 감싸듯 서서**
게이트 방향으로만 도주로를 열어두는 방식입니다.

```
push_dir = normalize(animal_pos - gate_pos)   # 게이트 → 동물 방향 (압박 기준축)

goal_L = animal_pos + rotate(push_dir, +flank_angle) * flank_radius  # 왼쪽 플랭크
goal_R = animal_pos + rotate(push_dir, -flank_angle) * flank_radius  # 오른쪽 플랭크
```

- `push_dir`: 게이트에서 동물을 향하는 단위벡터. "이 방향으로 동물이 가야 한다"는 기준.
- `flank_angle`(기본 75°): 두 로봇이 `push_dir` 기준으로 좌우 몇 도에 서 있을지.
  - 너무 작으면(예: 30°) 두 로봇이 너무 붙어서 옆으로 도주하는 동물을 막지 못함.
  - 너무 크면(예: 120°) 로봇이 동물 앞쪽에 서게 되어 오히려 게이트를 막아버림.
  - 75°는 도주로 차단과 압박 효율 사이의 실험적 균형점.
- `flank_radius`(기본 0.5m): 동물로부터 goal까지의 거리.
  - 너무 가까우면 동물이 도망가 버리고, 너무 멀면 압박이 안 됨.

### 왜 Strömbom 모델인가

규칙 기반으로 "동물 뒤 특정 거리에 붙어서 밀기"를 구현할 수도 있었지만,
Strömbom 모델은 **두 마리의 몰이꾼이 동시에 일관된 방향으로 협력**하는 수학적 근거를 제공합니다.
특히 "게이트 반대 방향 기준으로 좌우를 배정"하는 아이디어는 이 모델에서 직접 가져온 것입니다.

## 2. 좌우 배정: 이동 비용 최소화 + 히스테리시스

### 문제

매 tick마다 두 로봇 중 누가 LEFT, 누가 RIGHT를 맡을지 결정해야 합니다.
단순히 "robot8은 항상 LEFT"로 고정하면, 이미 오른쪽에 있는 robot8이 왼쪽으로 가야 하는 비효율이 발생합니다.

### 해결: 비용 최소화 배정

```
phi = atan2(animal - gate)   # 게이트→동물 각도

cost(sign=+1) = |robot8_각도 - (phi + flank_angle)| + |robot11_각도 - (phi - flank_angle)|
cost(sign=-1) = |robot8_각도 - (phi - flank_angle)| + |robot11_각도 - (phi + flank_angle)|
```

두 배정 경우 중 **총 이동 각도 합이 작은 쪽**을 선택합니다.
이미 왼쪽에 가까운 로봇을 왼쪽에 배정해서 불필요한 왕복을 줄입니다.

### 히스테리시스가 필수인 이유

비용 최소화만 적용하면, 동물이 경계선 부근에서 미세하게 움직일 때마다 최적 배정이 매 tick 뒤바뀝니다.
결과적으로 두 로봇이 0.5초마다 서로 반대 방향으로 이동하는 **채터링(chattering)** 이 발생합니다.

해결: 현재 배정과 반대 배정의 비용 차가 `swap_hysteresis_deg`(기본 20°) 이상일 때만 스왑합니다.
"확실히 더 유리할 때만 바꾼다"는 원칙으로 불필요한 왕복을 차단합니다.

## 3. 우선순위(Priority)를 명시적 토픽으로 발행한 이유

### 초기 설계의 문제

초기에는 `robot_controller`가 `avoid_goal` 토픽의 수신 여부로 자신의 우선순위를 **간접 추론**했습니다.
`avoid_goal`이 들어오면 LOW, 없으면 HIGH라는 가정이었습니다.

이 방식은 다음 상황에서 무너졌습니다:
- `avoid_goal`이 QoS latching으로 인해 이전 값을 다시 수신하는 경우
- SAFE_STOP 해제 직후 새 avoid_goal이 아직 도착하지 않은 타이밍

결과: 두 로봇이 동시에 자신을 HIGH로 판단하고 동시에 출발 → 재충돌.

### 해결: 명시적 `/robot_ns/priority` 토픽

`designator`가 매 tick마다 `/{ns}/priority` (String, TRANSIENT_LOCAL) 토픽에 `"HIGH"` 또는 `"LOW"`를 직접 발행합니다.
`robot_controller`는 이 값을 그대로 읽습니다. 추론 없음, 해석 없음.

**교훈**: 분산 시스템에서 상태를 "간접적으로 추론"하게 만들면 타이밍 의존 버그가 생깁니다. 상태는 명시적으로 발행하는 것이 맞습니다.

## 4. safe_dist를 0.2m → 0.5m로 올린 이유

### 문제

초기 `safe_dist = 0.2m`로 설정했을 때 SAFE_STOP이 거의 발동하지 않았습니다.

원인: Nav2 costmap의 `inflation_radius`(기본 0.35m)가 로봇 반경(약 0.17m)을 감싸는 팽창 영역을 만듭니다.
두 로봇의 팽창 영역이 겹치면 Nav2 플래너가 그 전에 경로를 우회하거나 아예 목표를 포기합니다.
즉, 실제 AMCL 거리가 0.2m가 되기 전에 Nav2가 먼저 멈춰버려서 SAFE_STOP 조건이 달성되지 않았습니다.

### 해결

`safe_dist = 0.5m`로 설정하면 Nav2 costmap 팽창 영역 밖에서도 SAFE_STOP이 발동합니다.
PVO의 예방 거리와도 일치시켜 두 안전장치가 같은 기준으로 동작하게 했습니다.

## 5. Homography에서 bottom-center를 사용하는 이유

탑뷰 카메라로 동물의 바닥(map) 좌표를 추출할 때, bounding box의 **중심점**이 아닌 **하단 중앙**을 사용합니다.

**이유**: 탑뷰가 완전한 수직 정사영(orthographic)이 아닌 이상, 동물의 높이(3D)가 카메라 투영 평면에서 위치 오차를 만듭니다.
동물의 발이 닿는 지점(bounding box 하단 중앙)이 실제 바닥 좌표에 가장 가깝고,
homography 행렬도 "바닥 평면의 점 ↔ 이미지의 점" 관계로 계산했으므로 하단 중앙 사용이 수학적으로 올바릅니다.
중심점을 쓰면 동물 키의 절반만큼 오차가 생길 수 있습니다.

## 6. ByteTrack을 ID 추적에 사용한 이유

YOLO 단독으로는 프레임마다 독립적인 bounding box만 나옵니다.
동물이 순간적으로 가려지거나(occlusion) 탐지가 끊기면, 재탐지 시 다른 개체로 인식할 수 있습니다.

ByteTrack은 탐지된 box들을 칼만 필터와 IoU 매칭으로 연결해 **동일 개체에 일관된 ID를 유지**합니다.
이 ID를 "잠그면(track lock)" 오탐 box나 다른 침입 개체가 들어와도 추적 타겟이 바뀌지 않습니다.

YOLO+ByteTrack 조합이 단순 탐지가 아닌 **신뢰할 수 있는 위치 추적기**로 동작하는 이유입니다.

## 7. 판단(designator)과 실행(robot_controller) 분리

전체 시스템을 하나의 노드에서 처리하는 대신, 두 역할을 명확히 분리했습니다.

| 역할 | 노드 | 하는 일 | 하지 않는 일 |
| --- | --- | --- | --- |
| 두뇌 | `designator` | goal 계산, 배정, PVO | 로봇 액션 직접 실행 |
| 손발 | `robot_controller` | undock/nav/dock 실행 | goal 재계산 |

**이유**:
- 로봇 1대가 죽거나 재시작해도 `designator`는 계속 goal을 계산하고 발행합니다.
- 두 PC에 각 역할을 분산했을 때 독립적으로 재시작/디버깅이 가능합니다.
- goal 계산 버그와 실행 버그가 완전히 분리되어 있어서 원인 추적이 훨씬 쉽습니다.

## 8. 타이머 기반 tick 루프에서 콜백 직접 처리를 피한 이유

각 토픽의 구독 콜백에서 바로 goal을 계산하는 대신, 콜백은 "최신 값 저장"만 하고
0.5초 타이머 `_tick`에서 일괄 판단합니다.

**이유**: ROS2에서 여러 구독 콜백이 동시에 들어올 때, 각 콜백 안에서 goal 계산이 실행되면
경합(race condition) 없이 일관된 상태를 보장하기 어렵습니다.
타이머 기반으로 한 시점에서 한 번만 판단하면 상태 일관성이 자동으로 보장됩니다.
또한 undock/dock 같은 블로킹 액션은 타이머 콜백이 아닌 별도 스레드에서 실행해야
타이머 자체가 멈추는 데드락을 피할 수 있습니다.
